A. 前端重放前清屏 + 稳定写入
replay received
reset xterm buffer
write chunks sequentially
stable final screen

A. 先修前端回放/重连污染

每次 replay 前重置 xterm，避免旧 buffer 与新 replay 叠加；并保证 replay chunks 按序写完后再滚到底。

优点

  • 改动小，最贴近“残影/重复”现象
  • 风险低，不影响 PTY 进程

缺点

  • 如果根因是后端 resize 污染，不能完全治本
  • 重连时可能短暂清屏再恢复
B. 后端 resize 防抖/会话尺寸锁定
tab 1: 120 cols
tab 2: 80 cols
aggregate resize
only stable changes reach PTY

B. 加强 resize 聚合与日志污染防护

限制异常尺寸、重复 resize、完成态 resize；必要时对多浏览器尺寸选择策略做更稳的锁定。

优点

  • 更治本，减少 Claude TUI 被反复重排
  • 对正在运行的终端更稳定

缺点

  • 需要更精确复现，避免影响正常自适应
  • 移动端/分屏可能有兼容风险
C. 回放用 headless xterm 规范化
raw PTY bytes
→ headless xterm replay
→ clean snapshot / canonical stream

C. 后端规范化输出后再给前端

把 ANSI 光标移动、覆盖式 TUI 输出先在后端归一化，再发送更干净的终端快照/日志。

优点

  • 长期最鲁棒，可改善历史日志阅读
  • 不依赖浏览器 xterm 状态

缺点

  • 实现复杂，可能超出本次 bugfix
  • 实时交互与 raw PTY 语义容易被破坏

我的初步推荐：先做 A，并补 B 中最保守的 guard；C 作为后续专项，不建议这次直接上。
